# Cross-Agent MCP Gateway

Generated by Codex for Claude Code configuration governance.

The `cross-agent-gateway` MCP server lets Claude Code and Codex request each other's programmable capabilities through a human-in-the-loop policy layer.

Claude Code loads the gateway from the plugin `.mcp.json`. Codex loads the same gateway from the workspace `.codex/config.toml`.

## Quickstart For Real Projects

1. Run `profile_project_risk` before changing a repository.
2. Run `init_project_collab` with `developer_safe` for normal code work, or `config_governance` for configuration governance work.
3. Run `analyze_requirement_clarity` or `request_clarification` before execution if the task may be ambiguous.
4. Use `create_plan` and `approve_plan` for user-approved scope.
5. Use `execute_approved_plan` or `implement_with_review` only after the task has scope, validation, rollback, and sensitive-area information.
6. Use `get_dashboard_brief` while work is active, and `get_dashboard_detail` when a human needs the full state.
7. Use `run_quality_gate` before closeout.

## Modes

- `cautious`: every cross-agent call requires explicit confirmation.
- `auto`: low-risk calls run automatically; sensitive calls require confirmation.
- `danger`: confirmation checks are disabled by the gateway.

Danger mode does not bypass platform, account, sandbox, network, or operating-system limits. It only disables this gateway's confirmation gate.

## Capability Model

- Codex agent calls use `codex mcp-server` and its `codex` / `codex-reply` tools.
- Claude agent calls use `claude --print`.
- Claude tool calls use `claude mcp serve` and proxy a named Claude MCP tool.
- Task collaboration uses local `reports/tasks/` archives.

The gateway can coordinate Claude Code CLI and Codex through programmable MCP/CLI surfaces. It does not directly operate a pre-existing Codex Desktop chat window. Codex Desktop can still participate as the user-facing workspace when it is configured to load this gateway for the trusted project; shared task archives, confirmations, audit events, and quality evidence remain the coordination record.

## Real Project Workflow

Start by profiling a project without reading or printing secret values:

```text
profile_project_risk
```

For a persistent project policy, run:

```text
init_project_collab
```

Supported presets:

- `developer_safe`
- `developer_fast`
- `config_governance`
- `sandbox_full_auto`

Project policies are read from the active project directory when a request supplies `projectDir` or `cwd`. This lets one repository run in `developer_safe` while another uses `config_governance` without sharing a single global risk posture.

Use `get_collab_status` before and after multi-agent work to see current mode, pending confirmations, active file locks, recent audit events, and a compact project risk profile.

Use `emergency_stop` whenever the collaboration feels unclear or risky. It switches to cautious mode, pauses new cross-agent dispatches, cancels pending calls, and releases locks.

## Complex Project Quality Flow

For complex project work, prefer the high-level workflow tools over raw `call_codex` / `call_claude_cli`:

1. `analyze_requirement_clarity` checks whether the task is specific enough to execute. If not, `request_clarification` returns the highest-impact missing questions.
2. `start_project_task` creates a draft task plan with scope, non-goals, validation, risk, and rollback expectations.
3. `build_project_context_pack` summarizes project structure, conventions, test hints, and risk.
4. `implement_with_review` prepares a Claude-led implementation dispatch with confirmation gates.
5. `ask_codex_to_review` requests read-only independent review after implementation evidence exists.
6. `record_run_evidence` attaches validation, review, changed-file, and rollback evidence to a resumable run.
7. `run_quality_gate` checks whether goal, validation, review, rollback, changed-file scope, and open-risk evidence are complete.
8. `summarize_final_result` produces a user-facing closeout summary.

`get_user_dashboard` gives the human a compact view of pending confirmations, file locks, project risk, and next best action.

## User-Centered Workflow Tools

The gateway now has a middle layer between raw cross-agent calls and fully manual task archives:

- `analyze_requirement_clarity` and `request_clarification` stop vague tasks before execution. They return a readiness status, confidence level, missing fields, safe defaults, and up to 3 questions.
- `start_collab_run`, `get_collab_run`, `advance_collab_run`, and `record_run_evidence` provide a resumable task state machine. Typical states are `clarifying`, `planned`, `approved`, `implementing`, `validating`, `codex_reviewing`, `fixing`, `final_gate`, `done`, and `canceled`.
- `create_plan`, `revise_plan`, `approve_plan`, and `execute_approved_plan` support plan-first work. Execution is not prepared until a saved plan is approved.
- `explain_pending_confirmation`, `approve_pending_call`, `deny_pending_call`, and `request_plan_change` turn HITL approvals into explicit user choices.
- `get_task_timeline`, `append_task_event`, and `summarize_task_progress` record a task timeline and next best action.
- `propose_changes`, `review_patch_plan`, `approve_apply`, and `apply_confirmed_changes` separate change preview from application. Applying an approved proposal still goes through the active confirmation policy.
- `select_quality_template` and `explain_quality_failures` make closeout checks visible and actionable.
- `explain_error`, `suggest_recovery`, `retry_last_step`, and `rollback_last_change` help the user recover from failures without hidden automatic reruns.
- `export_task_report`, `generate_pr_summary`, and `generate_handoff_summary` produce shareable reports for PRs, reviews, and agent handoffs.
- `recommend_agent_route`, `explain_agent_route`, and `override_agent_route` make routing transparent while preserving human control.
- `get_dashboard_brief` is a compact status view; `get_dashboard_detail` includes pending confirmations, locks, project risk, recent audit entries, and quality hints.

This layer is intentionally workflow-oriented rather than autonomy-oriented. It improves convenience and output quality while keeping risky execution behind approval gates.

## Requirement Clarification Gate

The gateway treats a task as not ready for formal execution when important details are missing, including:

- a specific goal,
- scope or affected files,
- success criteria,
- validation method,
- rollback expectation,
- and sensitive areas for configuration, authentication, payment, data migration, permission, or production-impacting work.

`implement_with_review` and `execute_approved_plan` run this gate before dispatch. If the gate fails, they return `needs_clarification` instead of calling another agent.

## Resumable Run State

Use `start_collab_run` for complex work that may span multiple agent turns. It writes a run record under gateway state and chooses `clarifying` when the requirement gate fails, otherwise `planned`.

Allowed transitions are intentionally narrow:

- `clarifying` -> `planned`, `blocked`
- `planned` -> `approved`, `blocked`
- `approved` -> `executing`, `blocked`
- `executing` -> `reviewing`, `validating`, `blocked`, `rolled_back`
- `reviewing` -> `executing`, `validating`, `blocked`
- `validating` -> `completed`, `executing`, `blocked`
- `blocked` -> `planned`, `rolled_back`

This keeps long tasks recoverable without pretending the gateway can infer human approval from conversation alone.

## Quality Evidence

`run_quality_gate` accepts structured evidence, including validation command results, review evidence, changed files, and allowed file scope. A validation record with a non-zero `exitCode` fails the gate. A changed file outside `allowedFiles` also fails the gate.

The quality gate is not a substitute for tests; it is a closeout checklist that makes missing evidence visible before the task is called done.

## Confirmation Flow

High-risk calls return:

```json
{
  "status": "pending_confirmation",
  "callId": "call-...",
  "requiredConfirmationText": "I, the human user, approve call-..."
}
```

The user or controlling client must call `confirm_cross_agent_call` with the exact text, then retry the original call with `confirmedCallId` and `confirmationText`.

Confirmations are bound to a SHA-256 hash of the target, prompt, cwd, requested capability, declared files, tools, commands, intent, trace ID, parent call ID, and depth. A confirmation for one pending call cannot be reused for a different request.

Claude tool calls include the requested Claude tool arguments in risk classification and confirmation binding, so a harmless-looking tool name cannot hide a risky command or write request.

`danger` mode uses the same bound confirmation flow. Calling `set_policy_mode` with `mode: "danger"` first returns a pending confirmation; the caller must confirm that call and retry the same request with `confirmedCallId` and `confirmationText`.

Gateway-local writes, such as creating task archives outside the current gateway project or initializing a project collaboration policy, also use the confirmation model when the active policy and risk classifier require it.

## Loop And Lock Guards

- Cross-agent calls accept `traceId`, `parentCallId`, and `depth`.
- Calls fail when `depth` exceeds `maxDelegationDepth`.
- `acquire_file_lock`, `release_file_lock`, and `list_file_locks` provide cooperative file locks for write-heavy workflows.
- Locks expire automatically after their TTL.
- Lock table updates are serialized with a lightweight lockfile, retried briefly under contention, and written atomically.
- State record IDs such as `callId`, `planId`, `proposalId`, and `taskId` are limited to safe filename characters. Unsafe IDs are rejected before reading or writing state JSON.

## Runtime State

The gateway stores policy, pending confirmations, and audit logs in:

1. `CODEX_CLAUDE_COLLAB_STATE_DIR`, if set.
2. `CLAUDE_PLUGIN_DATA`, if set.
3. `~/.claude/plugins/data/codex-claude-collab`.

Audit logs redact common token, secret, cookie, session, and private-key patterns before writing.

Set `CODEX_CLAUDE_COLLAB_CODEX_HOME` when Codex should use an isolated state directory instead of the default `~/.codex`. This is useful for disposable sandboxes or CI-style validation.

## Verification

Run the lightweight verifier from the Claude configuration workspace:

```bash
node plugins/codex-claude-collab/scripts/verify-mcp-gateway.mjs
```

Use `--deep` to also check Codex and Claude backend MCP handshakes:

```bash
node plugins/codex-claude-collab/scripts/verify-mcp-gateway.mjs --deep
```

## Safety Notes

- Keep `.claude` configuration governance in `cautious` mode unless intentionally testing automation.
- Do not use `danger` mode outside disposable sandboxes.
- The gateway cannot cryptographically prove that a confirmation came from a human; it enforces a structured confirmation step for cooperative clients and agents.
- Secrets should not be placed in prompts, task archives, or confirmations.
