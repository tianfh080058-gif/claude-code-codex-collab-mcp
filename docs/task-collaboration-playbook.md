# Task Collaboration Playbook

Generated by Codex for Claude Code configuration governance.

This playbook defines repeatable Codex and Claude Code workflows centered on task archives.

## Task Archive Lifecycle

1. Start: create a draft task archive with `/task-start <goal>`.
2. Clarify: use `/task-clarify <task-dir>` when important scope, success criteria, validation, rollback, routing, or sensitive boundaries are unclear.
3. Confirm: before formal execution, use `/task-confirm <task-dir>` after the user confirms details, boundaries, validation, rollback, primary agent, and reviewer.
4. Execute or hand off: use Claude Code directly or prepare Codex/Claude with `/task-ready-for-codex` or `/task-ready-for-claude` only after confirmation.
5. Review: create `REVIEW.md` with `/task-review-pack` or a readiness skill and ask the other agent to inspect.
6. Validate: record commands, manual checks, and unresolved gaps in `VALIDATION.md`.
7. Close out: summarize outcome, risks, decisions, and rollback with `/task-closeout`.

## Pre-Execution Confirmation Gate

Formal work must not begin until the user confirms:

- Goal and expected outcome.
- Success criteria.
- In-scope work.
- Out-of-scope work.
- Primary agent and reviewer.
- Files, systems, or repositories allowed to change.
- Files, systems, or repositories forbidden to change.
- Validation expectations.
- Rollback expectations.
- Sensitive areas that require separate approval.

If confirmation is missing, the agent may keep improving the task archive draft, but must not execute the task or ask another agent to execute it.

## Lightweight Automation

Use `/task-start <goal>` as the default entry point for non-trivial work. It should create `BRIEF.md`, `ROUTING.md`, `DECISIONS.md`, and `VALIDATION.md` with `Status: pending` and `Formal execution allowed: no`.

Use `/task-clarify <task-dir>` before `/task-confirm` when the archive has important uncertainty. It should ask at most 3 high-value questions, classify them as must-confirm, safe default, or non-blocking, and write `CLARIFICATION.md`.

Use `/task-ready-for-codex <task-dir>` or `/task-ready-for-claude <task-dir>` only after `/task-confirm <task-dir>` records explicit user confirmation. If confirmation is missing, readiness skills must list missing confirmation items and stop.

When using `/task-ready-for-codex`, expect `CODEX_PROMPT.md`. Paste that prompt into Codex so it receives the task role, scope, validation expectations, rollback expectations, and secret-handling rules without relying on chat memory.

## Workflow: Product Idea To PRD

- Primary: Claude Code.
- Reviewer: Codex optional for structure and completeness.
- Start with `/task-start <goal>`.
- Route as Claude-led unless the user asks for broad documentation governance.
- Expected archive focus: user, problem, MVP, non-goals, metrics, acceptance criteria.
- Handoff to Codex only for independent review or documentation refactor.

## Workflow: Implementation With Independent Review

- Primary: Claude Code.
- Reviewer: Codex.
- Claude should inspect existing code, implement minimal changes, and record validation.
- Use `/task-ready-for-codex <task-dir>` or `/task-review-pack <task-dir>` before asking Codex to review.
- Codex should focus on regressions, missing tests, risky permissions, and rollback clarity.

## Workflow: Configuration Governance

- Primary: Codex.
- Reviewer: Claude Code.
- Codex should inventory files, classify risk, back up before edits, patch minimally, and validate.
- Claude reviews runtime usefulness with `/claude-review-codex` or `/task-review-pack`.
- Do not edit `settings.local.json`, hooks, MCP, or secrets without separate confirmation.

## Workflow: Safety Audit

- Primary: Codex.
- Reviewer: Claude Code or `collaboration-reviewer`.
- Scope can include permissions, MCP configs, hooks, skills, agents, and memory files.
- Report secret-like findings without values.
- Prefer recommendations before changes.

## Workflow: Debugging Or Incident Triage

- Primary: Claude Code.
- Reviewer: Codex optional.
- Capture reproduction steps, observed behavior, expected behavior, logs summary, and attempted commands.
- Do not paste private logs or credentials into task archives.
- Use Codex review when the fix affects configuration, permissions, or multiple files.

## Workflow: Release Readiness

- Primary: Claude Code.
- Reviewer: Codex optional.
- Capture changed files, validation commands, manual QA, known risks, rollback, and monitoring.
- Use Codex when the release includes configuration, hooks, MCP, or agent behavior.

## Minimum Done Criteria

A task archive is ready to close when:

- `BRIEF.md` states goal and success criteria.
- `ROUTING.md` names primary and reviewer.
- `DECISIONS.md` records user confirmation before execution, or clearly states that execution never started.
- `VALIDATION.md` lists checks run and checks not run.
- `DECISIONS.md` records important tradeoffs.
- `HANDOFF.md` or `REVIEW.md` exists if another agent was involved.
- Secret values are absent.
- Rollback is documented for any mutation.
