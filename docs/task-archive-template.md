# Task Archives

Generated by Codex for Claude Code configuration governance.

Use `reports/tasks/<timestamp>-<slug>/` for non-trivial work that may involve Codex desktop and Claude Code CLI.

The lightweight default entry point is `/task-start <goal>`. It creates a draft archive only. Formal execution remains blocked until `/task-confirm <task-dir>` records explicit user confirmation.

## Required Files

Each task archive should contain:

- `BRIEF.md`: goal, context, success criteria, constraints.
- `ROUTING.md`: primary agent, reviewer, allowed scope, forbidden scope.
- `CLARIFICATION.md`: up to 3 high-value clarification questions, if needed.
- `CODEX_PROMPT.md`: ready-to-paste prompt for Codex, created by `/task-ready-for-codex` when appropriate.
- `HANDOFF.md`: request to another agent, if needed.
- `REVIEW.md`: review request or review result, if needed.
- `VALIDATION.md`: commands, manual checks, evidence, gaps.
- `DECISIONS.md`: user choices, tradeoffs, remaining questions.

`/task-start` creates at least:

- `BRIEF.md`
- `ROUTING.md`
- `DECISIONS.md`
- `VALIDATION.md`

`HANDOFF.md` and `REVIEW.md` should be created later only when needed.

`CLARIFICATION.md` should be created by `/task-clarify <task-dir>` when the draft archive has important uncertainty. It should separate:

- Must Confirm Before Execution
- Safe Default Assumptions
- Non-Blocking Follow-Up

It should ask at most 3 questions.

`CODEX_PROMPT.md` should be created by `/task-ready-for-codex <task-dir>` only after confirmation. It should be a paste-ready prompt telling Codex its role, scope, validation expectations, rollback expectations, and secret handling rules.

## Pre-Execution Confirmation

Before any agent formally starts implementation, configuration changes, handoff execution, or review-triggered fixes, `DECISIONS.md` should include:

```md
## User Confirmation Before Execution

- Status: confirmed / pending
- Confirmed goal:
- Confirmed success criteria:
- Confirmed in scope:
- Confirmed out of scope:
- Confirmed primary agent:
- Confirmed reviewer:
- Confirmed allowed files / systems:
- Confirmed forbidden files / systems:
- Confirmed validation:
- Confirmed rollback expectation:
- Sensitive areas requiring separate approval:
```

If `Status: pending`, the archive is draft-only. The agent may refine the archive but must not execute the task.

`/task-confirm` is the only lightweight workflow step that may change the confirmation state to:

```md
- Status: confirmed
- Formal execution allowed: yes
```

The confirmation record must contain actual confirmed details, not just a generic note that the user approved.

## Directory Naming

Use:

```text
reports/tasks/YYYYMMDD-HHMMSS-short-slug/
```

Example:

```text
reports/tasks/20260619-130000-login-ux/
```

## Privacy Rules

- Do not store secret values.
- Do not copy full chat transcripts.
- Do not include local private paths unless necessary for execution.
- Summarize logs instead of pasting private raw logs.
- Mark sensitive tasks as requiring separate user confirmation.
