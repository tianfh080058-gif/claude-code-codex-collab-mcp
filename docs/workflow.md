# Codex And Claude Code Workflow

Generated by Codex for Claude Code configuration governance.

## Purpose

This workflow lets Codex desktop and Claude Code CLI collaborate without creating an unsafe automatic loop. Codex is the configuration governance and review assistant. Claude Code CLI is the runtime development and validation assistant.

## Responsibility Split

Codex should:

- Audit Claude Code configuration.
- Propose and apply minimal, backed-up configuration changes when authorized.
- Maintain docs, reports, backup manifests, and rollback instructions.
- Review diffs, hooks, MCP configuration, permissions, skills, agents, and commands.

Claude Code should:

- Use its runtime configuration during CLI sessions.
- Work inside real projects and validate developer experience.
- Generate structured handoffs or review requests for Codex when explicitly asked.
- Review Codex-generated reports and diffs when explicitly asked.

## Authorization Rules

- No cross-agent task starts without an explicit user instruction.
- Before formal execution starts, the active agent must confirm task details and boundaries with the user.
- The confirmation must cover goal, success criteria, in-scope work, out-of-scope work, primary agent, reviewer, allowed files or systems, forbidden files or systems, validation expectations, rollback expectations, and sensitive areas.
- If the user does not confirm, the agent may create or update a draft task archive, but must not execute, hand off for execution, or modify files for the task.
- A handoff file is not approval to execute. It is a structured request that the user can pass to the other agent.
- Changes involving secrets, permissions, hooks, MCP, `settings.local.json`, deletion, broad file moves, or external services require separate confirmation.
- Neither agent should assume the other has access to hidden state, chat history, browser state, or credentials.

## File-Based Handoff

Use task archives for non-trivial work:

```text
reports/tasks/YYYYMMDD-HHMMSS-short-slug/
```

Each archive should contain:

- `BRIEF.md`
- `ROUTING.md`
- `CLARIFICATION.md`
- `CODEX_PROMPT.md`
- `HANDOFF.md`
- `REVIEW.md`
- `VALIDATION.md`
- `DECISIONS.md`

For small one-off requests, `reports/handoffs/` and `reports/reviews/` remain acceptable.

Before execution, `DECISIONS.md` should record an explicit confirmation entry:

```md
## User Confirmation Before Execution

- Status: confirmed / pending
- Confirmed scope:
- Confirmed boundaries:
- Confirmed primary agent:
- Confirmed reviewer:
- Confirmed validation:
- Confirmed rollback expectation:
```

Each handoff must include:

- From
- To
- User Approval
- Scope
- Allowed Files
- Forbidden Files
- Task
- Validation
- Rollback
- Risks
- Open Questions

Each review request must include:

- Review Target
- Changed Files
- Risk Focus
- Commands Already Run
- Secrets Policy
- Expected Output

## Task Archive Workflow

Use this workflow when work spans product intent, implementation, review, validation, or configuration governance.

1. Start a draft archive with `/task-start <goal>`.
2. Clarify missing requirements with `/task-clarify <task-dir>` when scope, success criteria, validation, rollback, routing, or sensitive boundaries are unclear.
3. Refine the brief or route with `/task-brief <task-dir>` or `/task-route <task-dir>` when needed.
4. Confirm task details and boundaries with `/task-confirm <task-dir>` before formal execution.
5. Execute directly only if the primary agent is the current agent and `Formal execution allowed: yes`.
6. Prepare Codex with `/task-ready-for-codex <task-dir>` when Codex should review or execute.
7. Prepare Claude Code with `/task-ready-for-claude <task-dir>` when Claude should review runtime behavior or continue implementation.
8. Ask the user to explicitly dispatch any handoff before the receiving agent acts.
9. Generate a review pack with `/task-review-pack <task-dir>` when the other agent should review.
10. Close out with `/task-closeout <task-dir>` after validation.

Task archives should store executable summaries, not full chat logs.
`/task-start` creates drafts only; it does not authorize execution or cross-agent dispatch.
`/task-clarify` asks at most 3 high-value questions and does not authorize execution.
`/task-ready-for-codex` should generate `CODEX_PROMPT.md`, a ready-to-paste prompt for Codex desktop.

## Safety Rules

- Do not print secret values. Report only file, type, risk, and recommendation.
- Prefer documentation and proposed diffs before changing runtime behavior.
- Create backups before editing existing config files.
- Keep Codex-specific governance rules out of `CLAUDE.md` except concise collaboration boundaries.
- Keep detailed Claude runtime preferences out of `AGENTS.md` unless quoted for audit context.
- Do not add automatic hooks that call or steer another agent.

## Recommended Collaboration Flows

### Codex Audits Claude Configuration

1. User asks Codex to audit or improve `.claude`.
2. Codex inventories files, classifies risk, and writes or updates `docs/CONFIG_AUDIT_REPORT.md`.
3. Codex proposes a patch plan.
4. After user approval, Codex backs up files, patches minimally, validates, and writes a change report.
5. User may ask Claude Code to review the Codex change report.

### Claude Requests Codex Review

1. User invokes `/task-ready-for-codex <task-dir>`, `/task-review-pack <task-dir>`, or `/codex-review <scope>` in Claude Code.
2. Claude writes or outputs a review request in `reports/reviews/` or `reports/tasks/<task-id>/REVIEW.md`.
3. User passes that file to Codex.
4. Codex reviews and reports findings, risks, and rollback recommendations.

### Claude Hands Work To Codex

1. User invokes `/task-ready-for-codex <task-dir>`, `/task-handoff <task-dir>`, or `/codex-handoff <task>` in Claude Code.
2. Claude writes or outputs a handoff in `reports/handoffs/` or `reports/tasks/<task-id>/HANDOFF.md`.
3. User confirms Codex should execute it.
4. Codex follows backup, validation, and rollback rules.

### Claude Reviews Codex Output

1. User invokes `/claude-review-codex <report-or-diff>` in Claude Code.
2. Claude checks scope, changed files, safety, validation, and rollback.
3. Claude reports whether the change is safe to keep and what needs human review.

## Rollback Standard

Every config modification should include:

- Backup path.
- Exact rollback command.
- New files created and how to remove them if needed.
- Validation command results.
